实现这些高级功能的关键在于不再仅仅把 Canvas 当作一个静态的图片显示区域，而是把它当作一个具有交互性的“工作台” (Workbench)。我们将这个插件的系统命名为 "Gemini Canvas Workbench"。以下是针对你提出的三个核心功能的详细 Canvas 实现思路。核心系统架构图 (Conceptual)这个架构图展示了信息如何在用户、Canvas 插件、Content 中提供一个 <input type="file">，用 Canvas 的 drawImage 将用户上传的参考图（比如梵高的《星空》）绘制到一个小的“参考区” Canvas。提取/发送风格： Canvas 本身不能直接做复杂的风格迁移 AI 运算（那需要庞大的模型）。我们的策略是利用 Gemini 的多模态能力。联动逻辑：用户在插件内上传参考图。插件通过 canvas.toBlob() 拿到图片数据。通过 Background Script 将其发送给 Content Script。Content Script 自动点击 Gemini 网页上的“上传图片”按钮，并将 Blob 数据“粘贴”进去。Content Script 同时在 Prompt 输入框中追加一句："请以这张图片的风格（[Style from attached image]）生成下一张图片..."2. 功能：手绘线稿转图片 (Sketch-to-Image)这个功能的关键是将用户的自由手绘转化为 AI 可理解的结构。Canvas 角色： 一个支持鼠标/触控交互的交互式画板。实现步骤：创建画布： 创建一个主 Canvas。监听交互： 监听 mousedown、mousemove、mouseup（或触控事件），在 Canvas 上使用 ctx.beginPath()、ctx.moveTo()、ctx.lineTo()、ctx.stroke() 绘制线条。联动逻辑：用户画完线稿（例如一个简单的火柴人）。插件提供一个“Link to Gemini”按钮。点击后，插件用 canvas.toDataURL('image/png') 导出 Base64 线稿图片。Content Script 将线稿上传到 Gemini，并在 Prompt 中自动填写："这是一张手绘线稿，请将其转化为一张高清的、写实的（[Real style, detailed from attached sketch]）图片..."3. 功能：图片局部修改 (Partial Modify / Inpainting)这个功能的关键是告诉 AI“哪里需要改”以及“改为什么”。Canvas 角色： 一个带蒙版（Mask）编辑功能的图像叠加层。实现步骤：加载原图： 插件抓取到 Gemini 生成的图片后，将其作为底层（Layer 0）绘制到 Canvas。创建蒙版层： 在原图上方创建一个半透明的蒙版层（Layer 1），默认透明。用户涂抹： 用户在插件内选择“遮罩工具”，在 Canvas 上涂抹（例如把生成的人像的“墨镜”区域涂黑）。这一步其实是在 Layer 1 上绘制不透明块。联动逻辑：插件点击“发送修改”。插件分别导出原图（Layer 0）和蒙版图（Layer 1，黑白二值图：涂抹区域为白，其余为黑）。Content Script 将两张图都上传给 Gemini（如果 Gemini 支持同时接收原图和蒙版），并在 Prompt 中填写修改指令，如："请将原图（[original image]）中蒙版区域（[mask image]）的墨镜，修改为一双（[modified area]）蓝色的眼睛..."技术难点与最终总结功能核心技术难点 (CORS & DOM)风格迁移Canvas drawImage + Gemini 多模态需要 Content Script 精准模拟图片上传操作，可能会因为 Gemini 网页更新而失效。线稿转图Canvas 鼠标/触控交互事件监听线稿图片不能太模糊，导出前可能需要做一次Canvas 的像素级增强（如锐化）。局部修改Canvas 多层合成 (globalCompositeOperation)最难实现。难点在于：1. 抓取原图时的 CORS 限制；2. 如何在 Prompt 极其有限的篇幅内清晰描述“原图、蒙版、指令”的三方关系。
